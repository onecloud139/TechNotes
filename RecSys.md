# 推荐系统整体架构

大部分工业级推荐系统可以抽象为 **三层结构**：

```
召回（Recall） → 排序（Ranking） → 重排（Re-Ranking）
```

# 召回层（Recall）

## 目标

召回层的核心目标是：

**从千万级、甚至上亿级的物品库中，快速筛选出几百个到几千个候选物品，交给后续排序层做更精细的打分。**

在真实推荐系统中，物品规模往往非常大，比如：

* 电商平台有上千万商品
* 短视频平台有海量视频内容
* 音乐平台有大量歌曲和歌单
* 新闻平台每天还在不断新增内容

如果每次用户打开首页时，都对全部物品做一次复杂排序，计算成本会高得无法接受，延迟也会非常大。因此推荐系统通常采用分阶段架构：

```text
召回（粗筛） → 排序（精排） → 重排（策略优化）
```

## 召回层的特点

### 1. 要速度快

召回层面向的是超大规模候选集合，因此必须在极短时间内完成筛选。工业系统里，召回通常要求在毫秒级完成，否则会直接影响用户体验。

### 2. 可以粗糙，但不能太差

召回层不是最终结果，它只是先找出一批“**可能有兴趣**”的候选集，所以允许不够精准。
但是它有一个底线：**不能漏掉真正重要的候选物品。** 因为一旦某个用户真正感兴趣的物品在召回阶段没有被选出来，那么后面的排序层再强，也没有机会把它排到前面。

## 召回层的基本逻辑

```text
用户进入推荐场景
→ 获取用户信息和上下文
→ 从不同召回通路中分别找候选物品
→ 合并多个召回结果
→ 去重、截断、生成候选集
→ 交给排序层
```

其中“不同召回通路”非常关键。工业界很少只依赖单一召回方式，而是通常采用 **多路召回（multi-recall）**。因为不同召回方法看到的是不同角度的“相关性”：

* 协同过滤擅长挖掘行为相似性
* 向量召回擅长学习深层兴趣表示
* 内容召回擅长处理冷启动和语义匹配
* 热门召回擅长做兜底和新用户场景

## 常见召回方法

### 1. 协同过滤召回

协同过滤（Collaborative Filtering）是推荐系统中最经典的一类召回方法。
它的核心思想不是直接理解物品内容，而是利用 **用户历史行为数据** 来发现关联关系。

> 不是看这个物品本身是什么，而是看“谁和谁行为相似”“什么和什么经常一起被喜欢”。

协同过滤通常分为两大类：

* **UserCF（基于用户的协同过滤）**
* **ItemCF（基于物品的协同过滤）**

#### UserCF

##### 基本思想

UserCF 的思路是：

* 找到和当前用户兴趣相似的用户
* 看这些相似用户喜欢了什么
* 把当前用户还没有接触过的物品推荐给他

```text
相似的人喜欢的东西，你也可能喜欢
```

##### 逻辑流程

UserCF 的一般流程如下：

1. 根据用户历史行为，构建用户-物品交互矩阵
   比如点击、收藏、购买、点赞等行为都可以作为兴趣信号。

2. 计算用户之间的相似度
   常见方法包括：

   * 余弦相似度
   * Jaccard 相似度
   * Pearson 相关系数

3. 找到与目标用户最相似的一批用户

4. 收集这些相似用户喜欢、但目标用户没有看过的物品

5. 按照相似度和行为强度加权，生成召回结果

##### 公式思想

```text
similarity(userA, userB)
```

推荐逻辑可以写成：

```text
userA 没看过 + userB 喜欢
```

##### 优点

* 实现简单，容易理解
* 直观地反映“相似用户”之间的兴趣迁移
* 在用户行为较丰富时效果可能不错

##### 缺点

* 用户规模特别大时，计算用户相似度成本高
* 用户兴趣变化快，相似用户关系不够稳定
* 对活跃度低的新用户效果差
* 工业场景中往往不如 ItemCF 稳定

所以在大规模推荐系统中，**UserCF 通常不是主力方法**。

#### ItemCF

##### 基本思想

ItemCF 的思路是：

* 找与用户历史喜欢物品相似的物品
* 把这些相似物品推荐给用户

```text
你喜欢过什么，就给你推荐和它相似的东西
```

##### 逻辑流程

1. 统计用户与物品的交互行为

2. 根据“共同被哪些用户喜欢”来计算物品之间的相似度

3. 对每个物品预先离线算出一组最相似物品列表

4. 在线推荐时，拿用户历史行为物品去查相似物品表

5. 合并多个相似物品，得到召回结果

##### 为什么工业界更常用 ItemCF

**ItemCF 更常见**，主要原因有几个：

1. **物品相似关系通常比用户相似关系更稳定**
   用户兴趣会变化，但“iPhone 和 AirPods 常一起购买”这种模式相对稳定。

2. **物品数量常常比用户数量少**
   电商平台可能有几千万用户，但商品数通常更可控，计算和存储相似度更现实。

3. **可解释性更强**
   比如“看了这个，所以推荐相似的那个”，用户和业务方都更容易理解。

4. **适合工程化实现**
   可以离线构建“物品-相似物品”索引，线上查表很快。

##### 优点

* 简单有效
* 稳定性较好
* 线上速度快
* 非常适合作为基础召回通路

##### 缺点

* 依赖历史共现数据
* 对新物品不友好
* 不能很好理解更深层语义关系
* 容易偏向已有热门物品

### 2. Embedding召回

它的核心思想是：

* 把用户表示成一个向量
* 把物品表示成一个向量
* 在向量空间中衡量用户和物品的相似程度
* 找出与用户向量最接近的一批物品

```text
用户 → 向量
物品 → 向量

向量相似度搜索
cosine similarity
inner product
```

#### 为什么要做向量化

传统协同过滤更多依赖显式共现关系，而向量召回希望学到更深层的兴趣表示：

* 用户虽然没看过某个物品，但兴趣模式和这个物品非常接近
* 两个物品虽然没有太多共现，但语义或潜在受众相似
* 用户兴趣是多维度的，向量可以更自然地表达复杂偏好

因此，Embedding 召回相比传统 CF，通常更有泛化能力。

#### 典型模型

##### Youtube DNN

YouTube 的候选生成模型是深度召回的重要代表。
它将用户历史观看、搜索、上下文等信息输入神经网络，输出用户兴趣表示，再与物品表示做匹配。

##### 双塔模型（Two Tower）

工业界最经典的向量召回架构之一就是 **双塔模型（Two Tower）**。

```text
user tower → user embedding
item tower → item embedding
```

用户侧和物品侧分别通过各自的网络映射到同一个向量空间中，然后通过向量相似度来判断匹配程度。

###### 双塔模型的直观理解

你可以把双塔模型理解成：

* 左边一座塔专门学习“用户是什么样的人”
* 右边一座塔专门学习“物品是什么样的内容”
* 学出来的两个向量如果距离近，说明这个用户大概率喜欢这个物品

用户塔可能输入：

* 用户画像
* 历史点击序列
* 历史购买类别
* 时间、场景、设备信息

物品塔可能输入：

* 商品ID
* 类别
* 品牌
* 文本描述
* 图像特征
* 标签信息

最终输出：

* user embedding
* item embedding

然后根据点积或余弦相似度做匹配。

###### 为什么双塔适合召回

因为它可以把 **物品向量提前离线算好并建索引**。
当用户来了，只需要：

1. 在线计算一个用户向量
2. 去向量索引库里查最相近的物品

这样就不需要对每个物品都单独过一次复杂模型，非常适合大规模召回。

###### ANN近邻搜索

如果有几千万甚至上亿个物品向量，仍然不能逐个算相似度。
所以在工程上会使用 **ANN（Approximate Nearest Neighbor，近似最近邻搜索）**。

> 在超大规模向量库中，快速找到和目标向量最相近的一批向量，而不必暴力遍历全部物品。

* **FAISS** 是常见的向量检索库
* **HNSW** 更准确说是一类近邻图索引算法
* **Milvus** 是向量数据库/检索系统

##### 向量召回的优点

* 表达能力强
* 泛化能力好
* 能捕捉更深层兴趣
* 适合大规模在线检索
* 是现代推荐系统主流方案

##### 向量召回的缺点

* 训练和部署复杂度更高
* 对特征工程和样本构造要求高
* 需要维护向量索引系统
* 调参和线上稳定性要求高

### 3. 内容召回（Content-based）

内容召回的思路是：

> 不依赖“别人怎么做”，而是直接根据物品本身的内容特征，找到和用户兴趣匹配的物品。

```text
用户喜欢 科幻电影
→ 推荐科幻电影
```

这类方法在冷启动场景中尤其重要，因为它不一定要求大量历史行为。

#### 内容召回依赖什么

内容召回通常会使用物品的各种属性信息，比如：

* 类别
* 标签
* 标题
* 文本描述
* 品牌
* 作者
* 主题
* 图像特征
* 音频特征

#### 0. 常见技术手段

```text
TF-IDF
BERT embedding
标签匹配
```

#### 1. 标签匹配

最简单的做法是给物品打标签，然后统计用户偏好的标签分布。
例如用户经常看“科幻”“太空”“悬疑”，那就优先召回包含这些标签的内容。

#### 2. TF-IDF

对于文本类内容，可以提取关键词并计算权重，用于做文本相似性匹配。

#### 3. BERT Embedding

更高级的方法是使用预训练语言模型，把标题、简介、评论等编码成语义向量，再做相似度搜索。
这样可以比简单关键词匹配更好地理解文本语义。

#### 内容召回的优点

* 对新物品友好
* 不强依赖历史行为
* 容易处理文本、图像、标签等多模态信息
* 在冷启动和语义理解方面很有价值

#### 内容召回的缺点

* 依赖内容质量和特征抽取效果
* 容易把用户限制在“相似内容”里，缺乏发现性
* 对行为层面的潜在关联挖掘不如协同过滤强


### 4. 热门召回

热门召回是推荐系统中最简单、但实际非常重要的一类召回方式。

```text
热榜
Trending
Top N
```

也就是，把当前平台整体最受欢迎、最活跃、最热门的一批内容召回出来。热门的定义可以有很多种，例如：

* 点击量最高
* 观看量最高
* 收藏量最高
* 购买量最高
* 互动率最高
* 最近一段时间增长最快

#### 1. 解决冷启动问题

这是它最经典的用途。
当用户是新用户，没有任何历史行为时，系统无法做个性化建模，这时最合理的策略往往就是先推荐热门内容。

* 新用户冷启动
* 新场景冷启动
* 无法获取用户画像时的推荐

#### 2. 作为兜底策略

即使是老用户，有时候其他召回通路也可能命中不足，或者系统状态异常。
此时热门召回可以作为稳定兜底，保证“至少有内容可推”。

#### 3. 提供平台公共兴趣

热门内容往往代表平台当前的公共关注焦点。
在新闻、短视频、社交平台中，这类内容本身就有很强消费价值。

#### 热门召回的优点

* 简单稳定
* 工程成本低
* 冷启动效果好
* 适合做兜底

#### 热门召回的缺点

* 个性化能力弱
* 容易过度集中到头部内容
* 可能造成“越热越推，越推越热”的马太效应

### 5. 多路召回：工业界的真实做法

真实推荐系统里，通常不会只用一种召回方法，而是会组合多种方法一起工作。

* ItemCF 擅长找“和历史行为相似”的物品，但对新物品不友好
* 内容召回擅长处理冷启动，但可能缺少行为信号
* 向量召回泛化强，但有时不够稳定或可解释
* 热门召回简单稳健，但个性化弱

```text
多路召回 → 合并 → 去重 → 截断 → 候选集
```

#### 多路召回的整体流程

```text
1. 用户请求到来
2. 根据用户特征和场景选择要启用哪些召回路
3. 各路召回并行返回候选
4. 对候选打上来源标记和原始分数
5. 融合、去重、截断
6. 输出给排序层
```

##### 1）并行执行

多路召回一般都是并行跑，不然延迟会太高。

##### 2）每路数量受控

不能让某一路无限制返回，否则会挤压其他路。

```text
ItemCF 200
向量召回 300
内容召回 150
热门召回 50
运营召回 30
```

##### 3）保留来源信息

每个候选通常要记录：

* 来自哪一路
* 原始召回分数是多少
* 是否来自多路命中
* 命中时的上下文

#### 融合策略

##### 策略1：直接拼接 + 去重

如果某个 item 重复出现，就只保留一份。

###### 优点

* 实现最简单
* 工程代价低
* 容易快速上线

###### 缺点

* 没有考虑不同通道质量差异
* 某一路可能候选太多，把其他路淹没
* 结果不稳定

##### 策略2：固定配额融合

```text
ItemCF 200
向量召回 200
内容召回 100
热门召回 50
运营召回 50
```

最后总共 600 个候选。

###### 优点

* 简单可控
* 可以保证每一路都有曝光机会
* 容易调试和观察效果

###### 缺点

* 配额是人工设的，不够灵活
* 某一路今天质量高也拿不到更多份额
* 某一路质量差也可能占坑

##### 策略3：按召回分数做统一排序

每一路召回都会返回一个原始分数，比如：

* itemcf 相似度
* 向量点积
* 文本相似度
* 热门值

先把它们转成统一尺度，再全局排序。

###### 难点

不同通路的分数**天然不可比**。

例如：

* ItemCF 分数可能在 0 到 1
* 向量点积可能在 -3 到 15
* 热门值可能是几百到几万
* 文本匹配可能是 BM25 分数

所以必须先做**分数归一化**。

常见方法：

* min-max normalization
* z-score normalization
* sigmoid 映射
* rank-based normalization

$$score_{final} = w_1 \cdot norm(score_{cf}) + w_2 \cdot norm(score_{vec}) + w_3 \cdot norm(score_{content}) + w_4 \cdot norm(score_{hot})$$

其中 (w_1,w_2,w_3,w_4) 是不同召回路的权重。

###### 优点

* 比固定配额更灵活
* 能利用每一路内部的置信度
* 可以更精细融合

###### 缺点

* 归一化很难做好
* 通道间分数漂移会影响稳定性
* 参数维护成本更高

##### 策略4：按名次融合

相比直接融合原始分数，很多系统更愿意融合“排名”。

* 在 ItemCF 中排第 3
* 在向量召回中排第 12
* 在内容召回中排第 5

可以用类似 Reciprocal Rank Fusion 的思路：

$$
score(i) = \sum_{r \in routes} \frac{w_r}{k + rank_r(i)}
$$

* 一个 item 在某一路排得越靠前，贡献越大
* 如果它在多路都出现，得分会叠加
* 多路共同命中的 item 往往更值得保留

###### 优点

* 不依赖不同通路原始分数的绝对尺度
* 对分数不可比问题更稳健
* 工程上很实用

###### 缺点

* 丢失了部分原始强度信息
* 需要选好每路权重和 rank 衰减参数


##### 策略5：多路命中加权

如果一个 item 被多路召回同时命中，通常说明它更可靠。

$$score(i) = base(i) + \alpha \cdot multi\_hit\_count(i)$$

* ItemCF + 向量同时命中，加 bonus
* 向量 + 内容同时命中，加 bonus
* 3 路以上命中，优先保留

###### 直觉

多路一致认为这个 item 相关，那它更可能真是好候选。

###### 注意

不能无脑偏好多路命中，否则会压制长尾和探索性内容。

##### 策略6：场景化融合

不同场景下，融合策略不一样。比如：

###### 新用户场景

没有行为数据，就应该提高：

* 热门召回
* 内容召回
* 新品召回

降低：

* ItemCF
* 深度行为召回

###### 老用户场景

行为丰富，就提高：

* ItemCF
* 向量召回
* 实时兴趣召回

###### 强时效场景

比如短视频、新闻流，要提高：

* 热点召回
* 实时召回
* 新内容召回

所以工业里常见做法是：

```text
先做用户/场景分桶
再配置不同召回路权重和配额
```

##### 策略7：学习式融合

更进一步，可以训练一个轻量模型来决定候选保留和融合分数。输入特征包括：

* 来自哪些召回路
* 每一路原始分数
* 每一路名次
* 是否多路命中
* 用户类型
* 物品类型
* 场景特征

输出一个融合分数。比如：

* LR
* GBDT
* 小型 MLP

这可以看成在排序前做一个轻量 pre-rank / merge rank。

###### 优点

* 自动学习不同通路的重要性
* 更适合复杂场景
* 融合效果通常更好

###### 缺点

* 工程复杂度更高
* 训练样本构造麻烦
* 维护成本更高

#### 去重

融合之前或之后都要去重。因为不同通路经常召回到同一个 item。

去重时一般会保留：

* 该 item 的所有来源路由
* 每一路对应分数
* 最优名次
* 是否多路命中
```text
item123:
  sources = [itemcf, vector, hot]
  cf_score = 0.82
  vec_score = 13.7
  hot_score = 5621
  best_rank = 2
  multi_hit = 3
```

这样做的原因是：

1. **不同方法覆盖不同类型的兴趣**
2. **单一路径容易漏召**
3. **多路融合能提高整体召回率**
4. **能兼顾个性化、冷启动、实时热点和稳定性**


# 二、排序层（Ranking）

召回之后可能有：

```
500 ~ 2000个候选
```

排序模型要预测：

```
用户点击概率
```

常见目标：

```
CTR = Click Through Rate
CVR = Conversion Rate
```

排序任务本质：

```
点击预测
```

---

## 排序模型发展

### 1 LR（逻辑回归）

最早推荐系统模型。

公式：

```
P(click) = sigmoid(w·x)
```

优点：

* 快
* 可解释

缺点：

* 只能学线性关系

---

### 2 GBDT + LR

工业经典结构：

```
GBDT → 特征组合
LR → 预测
```

Facebook曾经大量使用。

---

### 3 Deep Learning模型

现在主流。

例如：

#### Wide & Deep

Google提出。

结构：

```
Wide：记忆
Deep：泛化
```

解决：

```
稀疏特征
```

---

#### DeepFM

自动学习特征交叉。

结构：

```
FM + DNN
```

---

#### DIN（阿里）

用于电商推荐。

核心思想：

```
Attention
```

根据用户行为 **动态计算兴趣权重**。

例如：

用户行为：

```
手机壳
耳机
充电器
```

当前物品：

```
AirPods
```

模型会自动关注：

```
耳机
```

而不是：

```
手机壳
```

---

# 三、重排层（Re-ranking）

排序之后再做 **策略优化**。

目的：

* 多样性
* 新颖性
* 商业目标

例如：

```
避免全是同一类商品
```

可能策略：

### 1 多样性

不要：

```
10个手机壳
```

要：

```
手机壳 + 数据线 + 耳机
```

---

### 2 曝光控制

避免：

```
头部商品垄断
```

---

### 3 商业目标

结合：

```
点击率
转化率
广告收益
```

---

# 四、推荐系统核心数据

推荐系统离不开 **用户行为数据**：

常见行为：

```
曝光
点击
收藏
购买
停留时长
```

数据形式：

```
(user, item, action, time)
```

例如：

```
(123, AirPods, click, 2026-03-10)
```
